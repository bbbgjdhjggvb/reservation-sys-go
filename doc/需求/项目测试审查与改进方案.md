## 二、发现的问题

### 2.1 测试失败 —— 需要修复

**文件**: `service/reservation/middleware/ratelimit_test.go`

两个并发测试在覆盖率扫描时失败：

- `TestAllow_ConcurrentBurst` (line 351)
- `TestAllow_PipelineAtomicity` (line 397)

报错 `Elements should be the same type`，原因是 `assert.LessOrEqual` 比较 `int64(passCount)` 与 `int64(max)` 时类型推断冲突。修复方式：

```go
// 修改前
assert.LessOrEqual(t, passCount, max, ...)

// 修改后 —— 统一为 int64
assert.LessOrEqual(t, passCount, int64(max), ...)
```

### 2.2 代码清理 —— 需要删除

**文件**: `service/reservation/middleware/ratelimit_compare_test.go`

`AllowPipeline` 已被确认为不适合当前低阈值限流场景（500 并发 / 阈值 5 时多误拒 1 次，膨胀窗口峰值达 17）。对比测试已完成使命，应将 `AllowPipeline` 函数和对比测试文件一并删除，避免日后误用。

### 2.3 缺少集成测试 —— 需要补充

**文件**: `tests/api/reservation_api_test.go`

当前 `setupReservationAPI()` 未挂载 `RateLimitMiddleware`，无法验证限流中间件在真实路由中的装配正确性。需要新增 `setupReservationAPIWithRateLimit()` 及对应测试用例，验证：
- 限流触发时返回 429（handler 的 mock repo 不被调用）
- GET 路由不受限流影响

## 三、需要新增的测试

### 3.1 高优先级：安全关键代码

#### `pkg/jwt/` —— JWT 生成与解析

**风险**: 未测试。Token 签名/验证逻辑错误将导致认证失效，直接影响系统安全。

**建议新增文件**: `pkg/jwt/jwt_test.go`

**测试用例（表格驱动）**:

```
TestGenerateUserToken:
  - 正常生成 token，验证非空且可解析
  - 空 openid 生成 token

TestParseUserToken:
  - 解析有效 token，验证 claims 中 openid 一致
  - 解析过期 token，验证返回错误
  - 解析伪造 token（错误密钥），验证返回错误
  - 解析空字符串，验证返回错误

TestGenerateAdminToken:
  - 正常生成，验证 admin_id/username/role 可解析

TestParseAdminToken:
  - 解析有效 admin token
  - 解析过期 admin token
  - 使用用户密钥解析 admin token（跨类型攻击），验证失败

TestInitUserJWT_syncOnce:
  - 多次调用 InitUserJWT，验证仅首次生效
  - 空密钥时使用默认密钥
  - 零或负过期时间时保持默认24小时

TestInitAdminJWT_syncOnce:
  - 同上方用户 JWT 测试
```

## 四、需要改进的现有测试

### 4.1 缺少表格驱动测试

部分测试用例以独立 `t.Run` 编写但未使用表格驱动模式，参数化程度不够。例如：

**`service/reservation/handler_test.go`** —— Submit 参数校验测试分散在多个 `t.Run` 中，但每个用例的 body/期望码/期望消息可抽象为一个结构体切片，提高可读性和新增用例的效率。

### 4.2 缺少 t.Helper() 标记

**文件**: `service/reservation/handler_test.go` — `setupTestHandler`
**文件**: `tests/api/reservation_api_test.go` — `setupReservationAPI`, `generateTestToken`

这些辅助函数应添加 `t.Helper()`，使得测试失败时调用栈指向实际测试用例而非辅助函数。

### 4.3 缺少测试数据隔离

部分测试函数内的 mock 期望未设 `Times(1)`，如果代码路径变化导致 mock 未被调用或重复调用，不会报错。例如 `handler_test.go` 中部分 `EXPECT()` 调用缺少 `.Times(1)` 约束。

## 五、缺少的测试类型

整个项目当前**完全没有**以下测试类型，建议在关键路径上补充：

### 5.1 基准测试（Benchmark）

建议新增：

| 文件 | 测试 | 理由 |
|------|------|------|
| `pkg/jwt/jwt_test.go` | `BenchmarkGenerateUserToken` | 每次 API 请求都需解析 JWT |
| `pkg/reservationdb/repository_test.go` | `BenchmarkCreateOrderWithLock` | 核心写路径，含事务+行锁 |
| `service/reservation/middleware/ratelimit_test.go` | `BenchmarkAllow` | 每个请求都经过限流判断 |

### 5.2 模糊测试（Fuzz）

预约提交接口接收大量用户输入，应添加 fuzz 测试检测解析边界：

| 文件 | 测试 | 理由 |
|------|------|------|
| `service/reservation/dto_test.go` | `FuzzSubmitReqValidation` | 验证 JSON body 的极端输入不会 panic |

### 5.3 并发安全测试

已有个别并发测试（`TestAllow_ConcurrentBurst`），但缺少：

| 文件 | 测试 | 理由 |
|------|------|------|
| `pkg/reservationdb/repository_test.go` | `TestCreateOrderWithLock_Concurrent` | 验证 `SELECT FOR UPDATE` 行锁正确阻止并发双重预约 |

已有测试可加 `t.Parallel()` 提升执行效率（`ratelimit_test.go` 中的 5 个 Allow 测试相互独立，适合并行）。
