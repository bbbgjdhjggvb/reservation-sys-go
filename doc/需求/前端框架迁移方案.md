# 前端框架迁移方案

## 目标

将当前由 Gin `LoadHTMLGlob` 渲染的纯 HTML + vanilla JS 前端，迁移为 Vue 3 单页应用（SPA），实现前后端分离。

## 为什么选 Vue 3

| 维度 | Vue 3 优势（对本项目） |
|------|------|
| 模板语法 | 与当前 HTML 拼接思路一脉相承，迁移心智成本低 |
| 单文件组件 | `<template>` + `<script>` + `<style>` 天然对应现有页面结构 |
| 响应式系统 | `ref` / `reactive` 替代全局变量 + `innerHTML`，自动追踪状态变更 |
| 打包体积 | ~33KB gzip，微信内置浏览器加载更快 |
| Pinia 状态管理 | 官方方案，解决当前 authToken / selectedSlots 跨组件传递问题 |
| 中文生态 | 文档、社区、UI 库（Element Plus / Naive UI）中文支持极好 |
| Vite 构建 | 秒级热更新，开发体验远超 Go 模板重编译 |

### 现有架构

```
微信浏览器
    │
    ▼
nginx :80
    ├── /          → reservation:8081 (Gin 渲染 index.html)
    ├── /api/v2/*  → reservation:8081 (REST API)
    ├── /static/   → nginx 直接提供静态文件
    ├── /admin     → admin:8082 (Gin 渲染 admin 页面)
    └── /api/v3/*  → admin:8082 (REST API)
```

### 认证流程（当前）

```
1. 用户访问 / → Gateway OAuth 回调 → 签发 JWT
2. Gateway 302 重定向到 /?token=xxx
3. 前端 JS 从 URL 取 token → 存入 localStorage
4. 后续 API 请求带 Authorization: Bearer <token>
```

## 分阶段迁移计划

### 第一阶段：基础设施

1. **创建 frontend 目录**，初始化 pnpm workspace + Vite + Vue 3 + TypeScript
2. **配置 Vite proxy**：开发时 `/api/v2` → `localhost:8081`，`/api/v3` → `localhost:8082`，解决跨域
3. **引入 Tailwind CSS v4**：复用现有色板（荔枝红 `#8B0000`），定义主题 token
4. **搭建 API 客户端**：统一封装 fetch，自动注入 `Authorization` header，处理 401 过期跳转
5. **配置构建输出**：产物输出到 `service/reservation/frontend/dist/` 和 `service/admin/frontend/dist/`，由 Docker 构建时拷贝到 nginx 静态目录

### 第二阶段：reservation 用户端迁移

按组件从简单到复杂迁移：

| 优先级 | 组件 | 对应旧代码 | 说明 |
|------|------|------|------|
| 1 | `App.vue` + 路由 | `index.html` 结构 | 顶部导航栏 + `<router-view>` |
| 2 | `Toast.vue` | `showToast()` | 纯展示组件，无依赖 |
| 3 | `api/client.ts` | `apiRequest()` | API 封装 |
| 4 | `auth store` | token 读写 | 初始化时从 URL/localStorage 取 token |
| 5 | `AlumniAutocomplete.vue` | 模糊搜索 + 下拉 | 独立组件，可单独测试 |
| 6 | `SlotCell.vue` + `CalendarGrid.vue` | `renderCalendar()` | 核心组件，需对接 API |
| 7 | `ReserveForm.vue` | 表单 + 验证 | `v-model` 替代手动取值 |
| 8 | `ConfirmModal.vue` | `showConfirmModal()` | 用 props 控制显隐 |
| 9 | `ReserveView.vue` | 完整预约页 | 组合以上组件 |
| 10 | `MyOrdersView.vue` | `myorders.js` | 列表 + 取消操作 |

### 第三阶段：admin 管理员端迁移

| 优先级 | 组件 | 对应旧代码 | 说明 |
|------|------|------|------|
| 1 | `LoginView.vue` | `admin/index.html` | 登录表单 |

| 4 | `OrderTable.vue` | 订单列表渲染 | 筛选、分页 |
| 5 | `OrderDetail.vue` | 订单展开详情 | 审核记录时间线 |

### 第四阶段：后端适配 & 部署

1. **Go 端清理**：移除 `r.LoadHTMLGlob`，删除旧的 `frontend/*.html` 和 `.js` 文件
2. **nginx 配置调整**：`/` 和 `/admin` 从 `proxy_pass` 改为指向前端静态文件
3. **Docker 构建**：在 `scripts/build.sh` 中加入前端构建步骤（`pnpm build`），产物拷贝到 nginx 镜像
4. **认证回调适配**：Gateway 重定向 URL 由 `/?token=xxx` 改为前端路由 `/?token=xxx`（Vue Router 处理）
5. **冒烟测试**：走通完整的微信登录 → 预约 → 审核流程

## 关键技术决策

### 1. 认证流程（保持现有方案）

```
微信 OAuth → Gateway 回调 → 签发 JWT → 302 → 前端 /?token=xxx
                                              │
                                              ▼
                               Vue Router 入口 → store 提取 token
                               → localStorage 持久化
                               → API client 每次请求自动携带
```

不改为 SPA 登录页 + fetch 登录，因为微信 OAuth 必须走 Gateway 回调。这是最少的变更。

### 2. Tailwind CSS 复用

当前页面通过 CDN 引入 Tailwind。迁移后改为 Vite 构建时编译，按需生成 CSS（从几百KB CDN 降到 ~10KB 实际使用量）。荔枝红主题色定义为 Tailwind preset：

```js
// tailwind.config.js
theme: {
  extend: {
    colors: {
      primary: {        // 荔枝红
        50: '#fff5f5',
        100: '#ffe0e0',
        500: '#8B0000',
        700: '#660000',
        900: '#4a0000',
      }
    }
  }
}
```

### 3. 不引入 UI 组件库

当前页面量少（4页），引入 Element Plus / Naive UI 会增加 ~200KB+ 体积，性价比低。全部用 Vue + Tailwind 手写组件，保持轻量。

### 4. 状态管理 —— 用 Pinia，不用 provide/inject

原因：authToken 和 selectedSlots 需要跨多个视图/组件共享。Pinia 比 provide/inject 更可调试（Vue DevTools 支持），且复杂度低。

### 5. 构建产物部署位置

```
构建时：frontend/packages/reservation → dist/
Docker 构建：COPY dist/ → /usr/share/nginx/html/
                                ├── index.html
                                ├── assets/
                                │   ├── index-abc123.js
                                │   └── index-def456.css
                                ├── admin/
                                │   ├── index.html    (admin SPA)
                                │   └── assets/
                                └── static/           (保留的静态资源)
```

nginx 配置：
```nginx
# reservation SPA
location / {
    root /usr/share/nginx/html;
    try_files $uri $uri/ /index.html;   # SPA history mode
}

# admin SPA
location /admin {
    root /usr/share/nginx/html;
    try_files $uri $uri/ /admin/index.html;
}
```

## 风险与注意事项

1. **微信 OAuth 回调 URL 不变** —— 回调仍然走 Gateway，只是重定向终点从 Gin 模板变为前端 SPA 路由，Gateway 代码无需改动。

2. **兼容微信内置浏览器** —— Vue 3 支持到 iOS 10+ / Android 5+ 的 WebView，微信内置浏览器基于系统 WebView，完全兼容。Vite 默认构建目标为 `es2015`，需维持此设置（不要改成更老的 polyfill，微信浏览器不需要）。

3. **SEO 无影响** —— 所有页面均需微信授权后才能访问，不依赖搜索引擎收录。

4. **浏览器后退按钮** —— SPA 需要 `vue-router` 的 history mode 配合 nginx `try_files`，否则刷新 `/myorders` 会 404。

5. **迁移期间不中断现有功能** —— 建议在 `frontend/` 下并行开发，开发阶段用 Vite dev server 的 proxy 对接现有 Go API，全部完成后再切换 nginx 指向。

6. **文件体积增加** —— 新增 `frontend/` 目录（含 `node_modules`），需更新 `.gitignore` 添加 `node_modules/` 和 `dist/`。
