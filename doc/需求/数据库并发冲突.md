# 数据库并发冲突
## 什么是事务
事务是数据库保证操作 ACID 的最小工作单位。

事务可以并发执行。

---

## 事务并发冲突会导致什么事情
1. 脏读: 事务A读到事务B为提交的数据
```
# account = 100
WA(0 -> ac)      -> ARollback(100 -> ac)
    RB(ac = 0) -> 事务 B 拿着 ac = 0 去做其他事情
            
```
2. 不可重复读: 同一个事务内，两次读取同一行数据得到不同结果
```
# account = 100
RA(ac = 100) -> 拿着 ac == 100 去做事情
                                        RB(0 -> ac)
                                                        RA(ac=0)
```
3. 幻读：用一个事务内，两次查询返回的行数不同
```
RA(读到只有0行数据) -> insert
                RB(读到只有0行数据) -> insert
RA(读到两行数据，明明只能有一行)
```
4. 丢失更新: 两个事务同时读取同一行数据，各自修改后写入，最后一个写的覆盖掉前一个写的

---

## MySQL 的四种隔离级别
1. `READ UNCOMMITTED`: 可以读取未提交数据，什么东西都没限制，会发生所有问题
2. `RAED COMMITTED`: 只能读取提交了的数据，可以解决脏读
3. `PEPEATABLE READ(默认)`: 可重复读，可以解决脏读，不可重复读，丢失更新
4. `SERIALIZABLE`：串行化，所有问题都可解决

---

## MySQL 的锁机制
1. `Record Lock`: 给一条记录的索引加锁，其他事务不能对这一行修改删除
    1. 如果没有索引 `Record Lock` 会退化为表锁
2. `Gap Lock`: 在两条记录之间加锁，其他事务不能在这两条记录之间加入新的记录
3. `Next-Key Lock(Record Lock + Gap Loc)`: InnoDB 默认加锁方式
4. `Intention Lock`: InnoDB 在`UPDATE\INSERT\DELETE`时自动添加，当有其他事务要给这个表加锁的时候为等待阻塞，如果没有意向锁，B会逐行扫描有没有行锁，很慢

---

# 如何解决 Check-Then-Act问题
不允许读和写两个阶段分开，不允许其他事务中途插入

使用在`SELECT`语句后面加上`FOR UPDATE`，在查询的同时加入 gap lock(X)，直到事务提交，才将锁释放，不允许其他事务中途进行读写操作
